# Issue Label Taxonomy

This file documents the recommended Ghost Writer issue labels and intended use.

> This is documentation only. Apply labels manually in GitHub; do not automate label changes from this document.

## Core work-type labels

- **bug**: Incorrect current behavior that should be fixed.
- **docs**: Documentation-only work.
- **gameplay**: Player-facing mechanics, rules, progression, balance, or case flow changes.
- **accessibility**: Input, readability, contrast, audio alternatives, and other accessibility improvements.
- **tests**: Automated test coverage, test reliability, or verification improvements.
- **ci**: Build/check execution pipelines or automation reliability.
- **maintenance**: Refactors, cleanup, dependency-light tooling updates, and repository hygiene.

## Content and design labels

- **content**: Narrative text, clue writing, journal text, and authored non-mechanical content.
- **design**: System/puzzle/game design direction and tradeoff decisions.
- **playtest**: Human playtest planning, execution, and findings.

## Contributor onboarding and flow-control labels

- **good first issue**: Bounded tasks suitable for first-time contributors.
- **blocked**: Cannot proceed due to an unmet prerequisite.
- **human-only**: Requires direct human judgment/action (for example, live browser playtest verification or subjective design sign-off).

## Suggested usage rules

- Assign at least one work-type label to every issue.
- Add supporting labels (for example `playtest`, `blocked`) as needed.
- Keep labels current as issue scope evolves.
- Prefer opening a follow-up issue over overloading one issue with unrelated scope.
- Keep issue discussions and acceptance criteria up to date so the issue tracker remains the source of truth.
